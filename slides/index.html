Step 0 — 正規化

normalizer.normalize() 依序執行：

  1. NFC Unicode 正規化
  2. 全形 → 半形（１２３ → 123）
  3. 中文數字 → 阿拉伯數字（必須在民國年之前）
    • 位值：七十四 → 74
    • 單位:三月 → 3月、五號 → 5號
    • 連續：一一三 → 113、三三三五 → 3335
  4. 民國年 → 西元年（範圍 10–150）
  5. STT 語助詞壓縮：嗯嗯嗯嗯 → 嗯嗯（只壓縮 filler chars）
  6. 空白正規化

關鍵不變式：中文數字必須先於民國年轉換，否則「民國一一三年」無法識別

Speaker-aware / Diarization 降級

_apply_speaker_aware_masking 在無 diarization 時啟動：

  1. 問句 pattern 命中 → 之後 30 字內的 result score +0.15
  2. 答題 pattern 命中 → span 內的 result score +0.10

v4 冪等性修正（issue #2）：boost 收集後只套一次，不再 mutate RecognizerResult.score，避免重跑累加。 

q_boost[id(r)] = max(q_boost.get(id(r), 0.0), 0.15)
a_boost[id(r)] = max(a_boost.get(id(r), 0.0), 0.10)
new_r = copy.copy(r); new_r.score = min(1.0, r.score + total)

衝突為什麼會發生？

數字類撞類

OTP / PIN / CVV / 卡號 / 帳號 / 金額 都是數字串，regex 會互相命中。

多模型重疊

CKIP × spaCy NER × AddressRecognizer 可能在完全相同的 (start, end) 都抓 PERSON。

巢狀包含

AddressRecognizer 抓 「台北市忠孝東路100號」，CKIP 只抓 「台北市」 — 短 span 被長 span 包住。

ConflictResolver — 5 層決勝

  1. Step 0: Exact Duplicate Dedup — 相同 (start, end, entity_type) 保留 score 最高者；同分保留先出現者
  2. Contains — 嚴格包含時長者勝
  3. Risk Level — 部分重疊時 ENTITY_RISK_LEVEL 高者勝
  4. Span Length — 同風險時長者勝
  5. Priority Score — 最終 tiebreaker（公式見下頁）

關鍵不變式：kept 中任兩個 span 皆不重疊（已有 regression test 壓力測試 30 個隨機 span）

Priority Score 公式（v4 修正）

舊公式 priority 主導，與 Step 2a risk-level 決勝衝突。

priority_score = risk * 20.0       # 1..5 → 20..100 主導
               + priority * 0.1    # 25..100 → 2.5..10 次要
               + score * 5.0       # 0..1 → 0..5 第三
               + (2.0 if kw else 0.0)

保證：高 risk 無 keyword > 低 risk + keyword（20 > 2）。

舊問題實例：OTP(risk=5, pri=75) 被 PERSON(risk=4, pri=95) 蓋過 → 已修

conflict_log 輸出格式（v4）

每筆 entry：(winner_result, loser_result, reason_str)

reason 格式意義
EXACT_DUP:higher_score_winsStep 0 分數決勝
EXACT_DUP:first_winsStep 0 同分保留先出現
[R1] CONTAINS:longer_wins嚴格包含
[R2] CROSS_TYPE_SAME_SPAN:RISK_LEVEL:5>4同 span 跨 type 警示 + 原因
[R3] SPAN_LENGTH:12>8span 長度決勝
[R4] PRIORITY_SCORE:112.5>=98.4最終 priority tiebreak

[R{n}] = Step C 輪次索引（issue #7）
CROSS_TYPE_SAME_SPAN = 不同 type 但 span 完全相同（issue #5）

實例：三路爭「台北市忠孝東路100號」

#SourceTypeSpanScore
ACKIPLOCATION台北市 (4-7)0.85
BspaCyLOCATION台北市 (4-7)0.70
CAddressEnhancedLOCATION台北市忠孝東路100號 (4-16)0.80

Step 0：A vs B 為 exact dup → A 勝（0.85 > 0.70），寫 EXACT_DUP:higher_score_wins

Step C [R1]：C 嚴格包含 A → C 勝，寫 [R1] CONTAINS:longer_wins

最終保留：C → [ADDRESS]

Step 5 — 假名一致性（v4 固定 token）

PseudonymTracker.resolve() 一律回傳 TOKEN_MAP[entity_type]：

我叫王小明，我的卡號是1234567890123456
                ↓
我叫[NAME],我的卡號是[CARD]
v3 舊行為v4 新行為
第 1 個 → [NAME]_1、第 2 個 → [NAME]_2全部 → [NAME]

pseudonym_map 仍記錄原值 → token 對應（供 audit 追溯）。
字串替換按 start 倒序進行，避免位移錯亂。

Step 6 — 可用度標記 + Audit Log

usability_tag 判定優先序：

  1. LOW_AUDIO_QUALITY — ASR 信心 < 0.70
  2. FALLBACK_MODE — 無 diarization 但有問句降級介入
  3. NO_DIARIZATION — 無 diarization 也無問句
  4. DEGRADED_MASKING — 遮蔽密度 > 3/100 字
  5. USABLE — 正常

Audit CSV 欄位（config.AUDIT_FIELDNAMES）：

session_id, step, rule_triggered, entity_type, entity_subtype,
original_type_desc, start, end, score, token_applied,
conflict_resolved, diarization_available, usability_tag, timestamp

對話批次 mask_dialogue

處理 List[DialogueTurn]，跨 turn 共用同一 PseudonymTracker。

Diarization 判定（issue #3 修正）：

labeled_ratio = labeled_turns / total_turns
diarization_available = labeled_ratio >= 0.8

非 any(t.speaker for t in turns)，避免一通電話只有 1 句有 speaker 就誤判成整通都有 diarization。
