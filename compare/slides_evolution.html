為什麼要從 A 改成 B？

A 的六個痛點驅動了拔除 Presidio 的決定：

問題A 原版的現象
中文 PERSON NER 不準 spaCy zh_core_web_sm 漏掉「陳美玲」這類台灣常見姓名；反而被 VERIFICATION_ANSWER regex [\u4e00-\u9fff]{2,5} 亂切
語言過濾陷阱 Custom recognizer 預設 supported_language="en"，以 language="zh" 分析時全部被 Presidio 過濾；電話/OTP/ID/卡號全部漏遮，除非手動補 ="zh"
runtime 過重 Presidio + spaCy 啟動時載入 ~70 個 Python 模組，容器啟動 / 冷啟動時間長；升級 Presidio 會帶來語言/flag 等 API 不相容問題
Email 邊界溢出 Presidio 內建 EmailRecognizer 會把「我的Email是foo@example.com」整段抓成 EMAIL_ADDRESS
電話/帳號誤判 原版 TW_BANK_ACCOUNT regex \d{10,14} 沒排除 09 開頭，09xxxxxxxx 10 位手機會被當成銀行帳號
OTP context 不完整 原版 OTP_CONTEXT 缺「驗證碼」關鍵字，驗證碼是654321 這類關鍵 case 漏遮

B 做了什麼

  • 把中文 NER 來源從 spaCy 換成 CKIP bert-base-chinese-ner（繁中 + 台灣人名訓練）
  • 用 20 個 pure regex recognizer 取代 Presidio 的 PatternRecognizer — 同樣的 pattern / context boost 行為，但不依賴 Presidio
  • ConflictResolver 透過 duck-typed Span 繼續沿用
  • Email 自寫精準 regex，繞開 Presidio EmailRecognizer 的邊界問題
  • TW_BANK_ACCOUNT 補 negative lookahead (?!09\d{8})，解決電話誤判
  • OTP context 擴充，涵蓋「驗證碼」

代價：B 為了簡化把 A 的 normalize / PseudonymTracker / bank rules 全部丟掉，治理能力暫時缺席。

C 合併過程遇到的一個問題

normalize 把中文數字轉成阿拉伯數字後，暴露了 B 側 PIN regex 的脆弱：

案 1: 生日是民國七十四年五月一日
       → normalize → 生日是1985年5月1日
       → C 誤抓 → 生日是[PIN]年5月1日   ← 1985 被 PIN \d{4,6} 吃掉

案 4: 卡號末四碼是三三三五
       → normalize → 卡號末四碼是3335
       → C 誤抓 → 卡號末四碼是[PIN]     ← 3335 被 PIN 吃掉

根因：PIN pattern \d{4,6} score 0.30、無需 context 就被 resolver 保留 — 只要有 4 位數字就會 fire。

修正：require_context 硬約束

在 PurePatternRecognizer 加一個布林旗標：

class PurePatternRecognizer(Recognizer):
    def __init__(self, entity_type, patterns, context_keywords,
                 require_context: bool = False):
        ...
        self.require_context = require_context

    def analyze(self, text):
        for match in regex.finditer(text):
            kw = self._find_context_word(...)
            if self.require_context and kw is None:
                continue   # 沒有 context 關鍵字 → 直接 drop match
            ...

對三個「低分短數字」類別啟用：

  • PIN（\d{4,6}）
  • CVV（\d{3}）
  • BRANCH（\d{3,4}）

為什麼選這個、不選「拉高 threshold」：context 本來就是這類 pattern 唯一可靠的觸發條件。即使 threshold 拉到 0.55，只要 context boost +0.35 就會再次越過門檻 — 條件本質上還是 context。直接做成硬約束語意更清楚、效果更確定，也不需另外調 ConflictResolver。

結論與下一步

  • C 已具備生產可用的所有條件：辨識正確性繼承 B、治理能力繼承 A，且已修掉 normalize 暴露的 PIN 錯抓
  • 遷移建議：將 comparison/pipeline_c.py 收斂回主 repo 的 pipeline.py，逐步淘汰 original_file/
  • 效能：若流量敏感，優先走 CKIP GPU 推論；其次考慮 batch / 小模型
  • 監控：需加 audit log；上線後監測 normalize 是否又暴露其他 regex 類似的 context 依賴問題（例如 STAFF_ID、CAMPAIGN）
  • 治理：保留 PseudonymTracker 同時檢視 pseudonym_map 的治理規範（哪些欄位需要可逆、哪些純遮罩）

一句話總結：A → B 是「先拆掉 Presidio 驗證可行性」，B → C 是「把治理功能補回來」，PIN fix 是「合併時不能忽視 regex 脆弱點」。每一步都有明確動機與代價。
