兩側流程

A 原版

text
normalize（全形→半形、民國→西元）
Presidio AnalyzerEngine
├─ spaCy NER
└─ 17 custom recognizer
bank rules（條件式 AMOUNT + speaker boost）
ConflictResolver
PseudonymTracker → [NAME_1]、[CARD_1]

B 新版

text（不 normalize）
for rec in [...]: rec.analyze(text)
├─ 20 pure regex recognizer
└─ CKIPNer（bert-base）
ConflictResolver
固定 token → [NAME]、[CARD]
per-span replace

B 側 runtime 完全不 import presidio/spacy；ConflictResolver 透過 duck-typed Span 被兩側重用

A 勝出：normalizer

Case 1：全形逗號 → 半形

原文:   我叫王小明，卡號是1234567890123456
A 正規化: 我叫王小明,卡號是1234567890123456
A 遮罩:   我叫[NAME_1],卡號是[CARD_1]

Case 7：民國紀年 + 中文數字 → 西元

原文:   生日是民國七十四年五月一日
A 正規化: 生日是1985年5月1日
A 遮罩:   生日是1985年5月1日（DOB recognizer 未匹配，但可讀性還原了）
B 遮罩:   生日是民國七十四年五月一日（原文保留，無任何處理）

不過 case 7 A 雖然正規化了日期，DOB_SLASH 仍需 / 或 - 分隔符才會匹配 → 還是沒遮。等於只做了一半工作。

B 勝出 1：電話 vs 帳號

原文：我電話0912345678

A 原版

我電話[ACCOUNT_1]
→ TW_BANK_ACCOUNT('0912345678')

10 位數字被銀行帳號 regex 搶走

原版 TW_BANK_ACCOUNT pattern 是 \d{10,14} — 沒排除 09 開頭的手機號

ConflictResolver 在原版權重下 ACCOUNT 勝

B 新版

我電話[PHONE]
→ TW_PHONE('0912345678')

正確判定為手機

新版 bank account regex 是 (?!09\d{8})\d{10,14} — 用 negative lookahead 排除 09 開頭

同時優先級公式改為 risk-dominant

B 勝出 2：OTP context

原文：驗證碼是654321

A 原版

驗證碼是654321
(無任何實體偵測)

OTP regex \d{6} 有打到， 但原始 score 0.45 < threshold 0.50

需靠 context 加分才能過門檻。 原版 OTP_CONTEXT 沒有「驗證碼」關鍵字 → 沒加分 → 被濾掉

B 新版

驗證碼是[OTP]
→ OTP('654321')

B 的 pure regex OTP score 為 0.5， 且 context window boost 直接在 recognizer 內實作

OTP context 已更新為包含「驗證碼、OTP、驗證 code」

B 勝出 3：陳美玲

原文：母親的姓名是陳美玲

A 原版

母親[VERIFICATION_ANSWER][VERIFICATION_ANSWER]
→ VERIFICATION_ANSWER('的姓名是陳')
→ VERIFICATION_ANSWER('美玲')

兩個問題同時發生：

1. spaCy zh_core_web_sm 漏抓 PERSON「陳美玲」

2. VerificationAnswerRecognizer 的 regex [\u4e00-\u9fff]{2,5}|\d{6,8} 過度匹配 — 在「姓名是」trigger 後 30 字 window 內把任意 2-5 中文字都當答案

B 新版

母親的姓名是[NAME]
→ PERSON('陳美玲')

CKIP 直接抓到 PERSON

新版 VerificationAnswerRecognizer 的 regex 已改成 \d{6,8} — CJK 交給 CKIP 處理

B 勝出 4：Email span 精準度

原文：我的Email是foo@example.com

A 原版

[EMAIL]
→ EMAIL_ADDRESS('我的Email是foo@example.com')

整段都被當 EMAIL_ADDRESS

Presidio 內建 EmailRecognizer 與 context 互動過頭， 把「我的Email是」也吸進 span

B 新版

我的Email是[EMAIL]
→ EMAIL_ADDRESS('foo@example.com')

純 regex 精準抓 email 格式

[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}

結論 — 該往哪邊走？

  • 辨識正確性：B 在 4 個案例優於 A，A 只在 1 個案例因 normalizer 勝出 — B 的 recognizer 設計更接近生產可用
  • 效能：B CPU 慢 16.8×，但模型可切 GPU；
    A 的 presidio runtime ~73 模組是明確重量
  • 功能完整性：A 有 normalizer / pseudonym / 條件式 AMOUNT 等 — 這些都是 可移植的獨立模組，可以補回 B 側
  • 建議路線：以 B 為骨幹 + 把 A 的 normalizer、PseudonymTracker、bank rules 合併進來
