pipeline_b vs pipeline_c

合併版的實際增益

B 原本的 CKIP + 20 pure regex（不 import presidio / spacy）

C B 骨幹 + normalizer + PseudonymTracker + bank rules

Driver: comparison/compare_bc.py · 2026-04-18

為什麼做 C？

前一輪比較顯示原版 A 有三個 B 沒有的能力：

  • normalizer：全形/民國紀年/中文數字/STT 語助詞壓縮
  • PseudonymTracker：[NAME_1] / [NAME_2] — 同人一致、多人可區分
  • bank rules：條件式 AMOUNT（需與帳號/卡號並存才遮）+ speaker-aware boost

C 把這三件事加進 B 的骨幹，且保留 B 的優點（CKIP NER、Email 精準、電話不誤判、OTP context 完整、runtime 不 import presidio/spacy）。

實作：comparison/pipeline_c.py（重用 original_file.normalizer + original_file.pseudonym；bank rules 重新以 Span 介面實作，避免 presidio 相依）

Pipeline C 架構

B（原）

text
for rec in [...]: rec.analyze(text)
├─ 20 pure regex
└─ CKIPNer
ConflictResolver
固定 token
per-span replace

C（合併）

+ normalize()
for rec in [...]: rec.analyze(normalized)
├─ 20 pure regex
└─ CKIPNer
+ 條件式 AMOUNT（需 ACCOUNT/CARD 觸發）
+ speaker-aware boost（fallback）
ConflictResolver
+ PseudonymTracker → [NAME_1]、[CARD_1]
per-span replace

10 個針對合併效果設計的案例

  1. 全形/民國年：生日是民國七十四年五月一日
  2. 全形逗號：我叫王小明，卡號是1234567890123456
  3. STT 重複：嗯嗯嗯嗯我的卡號是1234567890123456
  4. 中文數字：卡號末四碼是三三三五
  5. 同人多次：我叫王小明，我就是王小明
  6. 多人區分：王小明跟陳美玲一起來分行
  7. 一般金額：今天氣溫30度，油價30元
  8. 交易金額：幫我轉帳500元到帳號123456789012
  9. 條件式 AMOUNT：這張卡號1234567890123456扣了1000元
  10. 電話 vs 帳號：我電話0912345678

C 勝 1：STT filler 壓縮

原文：嗯嗯嗯嗯我的卡號是1234567890123456

B

嗯嗯嗯嗯我的卡號是[CARD]

STT 語助詞保留，下游分析器得處理噪音

C

normalize → 嗯嗯我的卡號是1234567890123456
mask     → 嗯嗯我的卡號是[CARD_1]

3+ 個重複語助詞壓縮成 2 個

規則：_STT_FILLER_CHARS = "啊喔嗯欸哦呢吧吼哈呀哎唉那"，只壓語助詞，不動一般中文字

C 勝 2：Pseudonym 同值一致

原文：我叫王小明，我就是王小明

B

我叫[NAME]，我就是[NAME]

兩次出現 token 相同，但無從確認是否為同一人

C

我叫[NAME_1],我就是[NAME_1]
pseudonym_map:
  PERSON: {'王小明': '[NAME_1]'}

tracker 記住 原值→token 的對應

審計時可回推「同一人被說了幾次」

C 勝 3：多人可區分

原文：王小明跟陳美玲一起來分行

B

[NAME]跟[NAME]一起來分行

兩個 NAME 混在一起，無法辨識是兩個不同人

C

[NAME_1]跟[NAME_2]一起來分行
pseudonym_map:
  PERSON:
    '王小明': '[NAME_1]'
    '陳美玲': '[NAME_2]'

不同原值 → 不同編號

整通對話可持續追蹤多方實體

C 勝 4：條件式 AMOUNT

原文：今天氣溫30度，油價30元

B

今天氣溫[AMOUNT]度，油價[AMOUNT]

「30 度」「30 元」都被當 AMOUNT 遮

一般日常金額被過度遮罩 → 語料可讀性下降

C

今天氣溫30度,油價30元（無遮）

無 ACCOUNT / CARD 觸發 → AMOUNT 被 drop

規則：AMOUNT_TRIGGER_ENTITIES = {TW_BANK_ACCOUNT, TW_CREDIT_CARD}；proximity 60 字內

反之，case 9「卡號⋯扣了1000元」AMOUNT 保留

Regression：normalize 撞到 PIN regex

Case 1：生日是民國七十四年五月一日

C 正規化: 生日是1985年5月1日
C 遮罩:   生日是[PIN]年5月1日  ← 「1985」被當 PIN

Case 4：卡號末四碼是三三三五

C 正規化: 卡號末四碼是3335
C 遮罩:   卡號末四碼是[PIN]  ← 「3335」被當 PIN

根因：PIN recognizer pattern 是 \d{4,6} score 0.30，無需 context 就達 threshold 後 resolver 會保留。normalize 把中文數字轉成 4 位數字後就命中 PIN pattern。

修正方向（任選）：

  • 提高 PIN 門檻（0.30 → 0.55），要求 PIN context 關鍵字才能過
  • DOB recognizer 加入 \d{4}年\d{1,2}月\d{1,2}日 pattern，讓 1985年5月1日 優先被 DOB 吃
  • ConflictResolver 層級加入「PIN 需 context」硬約束

結論

  • 6 個案例 C 明顯優於 B — 涵蓋 normalize / pseudonym / bank rules 三大類能力
  • 1 個案例一致 — CKIP + Email / 電話 / OTP 等核心能力都被沿用
  • 2 個案例 C 引入 regression — normalize 暴露出 PIN regex (\d{4,6} @0.30) 的 context 依賴不足
  • 下一步：PIN threshold 拉高到 0.55 並強制需要 context，同時 DOB recognizer 補 \d{4}年\d{1,2}月\d{1,2}日
  • 完成上述修正後，C 可以直接取代 B 作為主 pipeline
